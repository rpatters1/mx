# mx::api Feature Audit

Generated by the `api-feature-audit` skill. Compares three views of the MusicXML feature surface:
the spec (approximated by `data/synthetic/`), the wild corpus (`data/` minus synthetic), and what
`mx::api` actually exposes (`data/api.features.xml`).

Corpus this run: 829 files (440 wild, 389 synthetic); 442 elements (326 used in the wild, 116
synthetic-only). `mx::api` audit: 318 features recorded (188 full, 63 partial, 67 none).

## How to read the severity

- BUG = silent data loss: a value `mx::core` (and the spec) can hold that `mx::api` maps to a
  fallback, so it is dropped on read and cannot be written back. Worst failure mode.
- GAP = an element/attribute `mx::api` does not model. Often by design; a gap in a heavily-used
  wild feature is a candidate to add.

Honest caveat up front: every enum BUG below is currently triggered only by the synthetic
(spec-coverage) corpus or by the core/XSD alone. None of the missing enum values appears in any wild
file in `data/`. They are real correctness bugs (the spec allows the value, a real-world file could
carry it), but the corpus does not exercise them today. That lowers urgency, not correctness. The
HIGH-USE GAPS in section 2, by contrast, are hit by hundreds of wild files right now.

Implementation path for everything here is the `add-mx-api-feature` skill (the two-sided
`mx::api` + `mx::impl` flow). Do not implement from this report.

## 1. BUG FIXES (enum members silently dropped)

All 30 `Converter::EnumMap` tables in `src/private/mx/impl/Converter.cpp` were diffed against their
generated `mx::core` enums. 9 tables drop core members. Within this section, the cheapest and
safest are the three where the `mx::api` enum already has the member and only the Converter table
row is missing (one-line, pure correctness, no api header change):

### 1a. One-line table fixes (api enum already has the member)

| Core value dropped | api enum (has slot) | Table / fallback | Wild? |
|--------------------|---------------------|------------------|-------|
| `AccidentalValue::sharpSharp` | `api::Accidental::sharpSharp` | `accidentalMap` Converter.cpp:78; fallback `Accidental::none` (`:1330`) | no |
| `NoteheadValue::faUp` | `api::Notehead::faUp` | `noteheadMap` Converter.cpp:42; fallback `Notehead::none` (`:1310`) | no |
| `SoundID::windReedClarinetPiccolo` | `api::SoundID::windReedClarinetPiccolo` | `instrumentMap` Converter.cpp:363; fallback `SoundID::unspecified` (`:1586`) | no |

`sharpSharp` is the most worth fixing of the three: a `<accidental>sharp-sharp</accidental>`
currently reads back as no accidental at all (`Accidental::none`), a clean visible data loss, and
the fix is a single map row. Effort: trivial each.

### 1b. Two-sided fixes (add an api enum member, then the table row)

Ranked by MusicXML version (3.1 additions first; they have synthetic coverage) and by how plausible
the value is in real music.

1. Dynamics `n`, `pf`, `sfzp` (MusicXML 3.1). `core::DynamicsChoice::Kind` has them;
   `api::MarkType` dynamics section stops at `fz`; `dynamicsMap` (Converter.cpp:170) omits them;
   fallback `MarkType::unspecified` (`:1445`). Synthetic: `n.3.1.xml`, `pf.3.1.xml`,
   `sfzp.3.1.xml`. This is the canonical example in the skill overview. Add three `api::MarkType`
   members + three rows. Effort: small.
2. Articulation `soft-accent` (3.1). `core::ArticulationsChoice::Kind::softAccent`; no `MarkType`
   member; `articulationsMap` (Converter.cpp:151); fallback unspecified (`:1425`). Effort: small.
3. Ornaments `haydn`, `inverted-vertical-turn` (3.1). `core::OrnamentsGroupChoice::Kind`;
   `api::MarkType` has `verticalTurn` but not these; `ornamentsMap` (Converter.cpp:197); fallback
   unspecified (`:1461`). Effort: small.
4. Technical simple marks `brass-bend`, `flip`, `smear`, `open`, `half-muted`, `harmon-mute`,
   `golpe` (3.1-4.0). `core::TechnicalChoice::Kind`; no `MarkType` members; `technicalMarkMap`
   (Converter.cpp:266); fallback unspecified (`:1481`). These carry no payload, so each is a clean
   enum + row. Effort: small.
5. Fermata 4.0 shapes `double-angled`, `double-square`, `double-dot`, `half-curve`, `curlew`.
   `core::FermataShape`; `api::MarkType` fermata section stops at `square`; `fermataMap`
   (Converter.cpp:356); fallback unspecified (`:1576`). Not in the synthetic corpus; core/XSD only.
   Effort: small.
6. Accidental and accidental-mark microtonal arrow/double variants `double-sharp-down`,
   `double-sharp-up`, `flat-flat-down`, `flat-flat-up`, `arrow-down`, `arrow-up` (plus the `other`
   catch-all). Affects both `accidentalMap` (-> `api::Accidental`, Converter.cpp:78) and
   `accidentalMarkMap` (-> `api::MarkType`, Converter.cpp:213). Rare; no corpus file. Effort:
   small-medium (two enums).
7. Notehead `circled` (4.0) and the `other` catch-all. `noteheadMap` (Converter.cpp:42). Effort:
   small.
8. Instrument-sound 4.0 SoundIDs `drum-tabor`, `drum-tamborim`,
   `pitched-percussion-handchimes`, `pluck-cavaquinho`, `wind-flutes-whistle-tin-c`,
   `wind-reed-clarinet-d`, `wind-reed-clarinet-g`. `instrumentMap` (Converter.cpp:363). Very rare.
   Effort: small but low value.

### 1c. Structural data loss (bigger than an enum row)

Technical `fingering`, `pluck`, `hammer-on`, `pull-off`, `bend`, `tap` are deliberately commented
out of `technicalMarkMap` (Converter.cpp:266) because they carry payloads a flat `MarkType` cannot
hold (text, start/stop, bend-alter/pre-bend/release). They fall back to `unspecified` and are lost.
Fixing them is not a one-line enum change; each needs an api data type. `fingering` (wild=8) is the
most common. Treat as a feature request, not a quick bug fix.

### Not a bug (verified)

`wedgeMap` (Converter.cpp:312) omits `WedgeType::stop` and `continue`, but that is by design:
`DirectionReader::parseWedge` (DirectionReader.cpp:436) branches on `WedgeType::Tag::stop` into
`api::WedgeStop` vs `api::WedgeStart` structurally, so stop is not lost. Minor nuance: a
`type="continue"` wedge is read as a start. Not data loss.

## 2. HIGH-USE GAPS (common in the wild, not fully modeled)

Ranked by wild-file count. These are hit by real files today.

1. `<print>` -- wild 165. partial, write-biased. Layout is written per measure
   (`MeasureWriter`), but the per-measure read is a stub: `MeasureReader::parsePrint` is a no-op;
   system/page info is reconstructed at score level (`ScoreReader::scanForSystemInfo` /
   `scanForPageInfo`). `new-system`/`new-page` survive; many `<print>` attributes are lost on read.
   Effort: medium.
2. `<credit>` family -- wild 142. partial. Credits with `<credit-words>` round-trip via
   `PageTextData` (`PageTextFunctions.cpp`), but: `<credit-image>` is not modeled; a `<credit>`
   with no `<credit-words>` is dropped (PageTextFunctions.cpp:53-61); multiple `<credit-type>` per
   credit collapse to the first (`:76-78`). Effort: medium.
3. `<sound>` -- wild 141. none. The element is dropped entirely:
   `MeasureReader::parseSound` is `MX_UNUSED` (MeasureReader.cpp:618) and `DirectionData.h:96`
   carries a `TODO - sound element`. Tempo/dynamics/segno changes encoded on `<sound>` (not on a
   `<direction>`) vanish. Effort: medium; high leverage given ubiquity.
4. Defaults fonts `<word-font>` (wild 129), `<lyric-font>` (109), `<music-font>` (91) -- none.
   Not referenced anywhere in `DefaultsData` or the impl; dropped on read and write. `FontData`
   exists for inline fonts, so the data type is mostly there; the gap is wiring them into
   defaults. Effort: small-medium.
5. `<figured-bass>` -- wild 3. partial and lossy. Read as plain text into a `WordsData` direction
   (`MeasureReader::parseFiguredBass` cpp:580); never written back as `<figured-bass>`, so it is
   lost on round-trip. Effort: medium (needs an api type).
6. Harmony extras `<inversion>` (wild 1), `<function>`, `<numeral>` (4.0) -- none. Note that the
   rest of harmony (root, kind, bass, degree, frame) is fully modeled via `ChordData`, so this is
   a narrow top-up, not a new subsystem. Effort: small each.

Lower-use gaps worth noting but not urgent: `staff-details` sub-elements beyond staff-lines
(capo, staff-type), `measure-style` beat-repeat/measure-repeat/slash, non-traditional key
(`key-accidental`/`key-octave`), `<percussion>` and all its children (synthetic-only), `<grouping>`,
`<harp-pedals>`, `<scordatura>`, performance/listen elements (mostly 4.0, synthetic-only).

## 3. ROUND-TRIP CLOSERS (cheap fixes, outsized round-trip impact)

- `midi-name` is read but never written: `PartReader.cpp:294` sets `MidiData.name`,
  `PartWriter.cpp` has no corresponding write. One-sided -> breaks round-trip for any file with a
  MIDI instrument name. Fix is a single writer addition. Effort: trivial.
- The three section-1a one-line table rows (`sharpSharp`, `faUp`, `windReedClarinetPiccolo`) are
  near-zero-cost correctness wins; ship them with any nearby change.
- Highest structural leverage for raising the full-round-trip rate of the wild corpus:
  `<sound>` (141 files), defaults fonts (~130 files), and `<credit-image>` / no-credit-words
  credits. A file that is otherwise fully supported but carries a `<sound>` tempo marking or a
  defaults `<word-font>` currently fails to round-trip on that alone. (Exact per-file closure sets
  were not computed this run; these are the dominant single-feature blockers by frequency.)

## Method notes

- Enum diff: `src/private/mx/impl/Converter.cpp` parsed for all `EnumMap` tables; each table's
  core-side keys diffed against the `enum class Tag`/`Kind` in the matching
  `src/private/mx/core/generated/<Type>.h`. Comments stripped so deliberately-commented rows count
  as omissions. All 30 tables checked; 21 are complete (step, duration, beam, clef, bool, valign,
  halign, css, fontStyle, fontWeight, measureNumbering, stem, lineType, barline, lineStop, bracket,
  placement, barlinePlacement, ending, kind/ChordKind, wedge-by-design).
- Element support: corpus-first from `data/corpus.xml`. Harmony, header/identification, layout, and
  instrument/MIDI clusters were verified directly against `mx::impl` readers and writers (file:line
  in `data/api.features.xml`). Core note/notation/direction/measure support is asserted from the
  `mx::api` data model plus reader/writer presence; entries inferred rather than line-traced are
  marked "suspected" in the notes.
- Wild-vs-synthetic: every missing enum value was grepped across `data/` (wild and synthetic
  separately) to set the "Wild?" column.
